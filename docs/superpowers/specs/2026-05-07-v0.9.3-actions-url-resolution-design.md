## Contexto

A re-validação manual de v0.9.2 expôs um bug remanescente:

- **BUG-VAL-001** ✅ paginação após mudar perPage — passou
- **BUG-VAL-002** 🟡 row actions — botões Edit/Delete renderizam mas dispatch retorna **404**
- **BUG-VAL-003** 🟡 bulk actions — botão renderiza mas dispatch retorna **404**
- **BUG-VAL-004** ✅ clear search — passou (search global Cmd+K é feature separada, fora deste scope)

A v0.9.2 corrigiu o **render** dos botões. O **dispatch** continua quebrado por causa diferente, agora no PHP.

## Diagnóstico técnico

### Estado actual de `Actions::edit()/delete()/view()/restore()/deleteBulk()`

`packages/actions/src/Actions.php` declara estas factory functions sem chamar `->url(...)` nem `->action(...)`:

```php
public static function edit(): RowAction
{
    return RowAction::make('edit')
        ->icon('pencil')
        ->color(Action::COLOR_PRIMARY)
        ->variant(Action::VARIANT_GHOST);
    // sem ->url() ou ->action()
}
```

`Action` é XOR entre `action(Closure)` callback ou `url(string|Closure, $method)` link. Quando ambos são null, `Action::toArray()` filtra o `url` (devolve null), `method` fica `POST` default. Frontend (`invokeAction` em `ArqelIndexPage.tsx`) cai no fallback `/arqel-dev/actions/${action.name}` que **não tem rota registada** → 404.

### Convenção esperada do framework

Inspecção das rotas registadas em `arqel-test` (Laravel `route:list`):

```
GET    admin/{resource}             arqel.resources.index
POST   admin/{resource}             arqel.resources.store
GET    admin/{resource}/create      arqel.resources.create
GET    admin/{resource}/{id}        arqel.resources.show
PUT    admin/{resource}/{id}        arqel.resources.update
PATCH  admin/{resource}/{id}        arqel.resources.<patch>
DELETE admin/{resource}/{id}        arqel.resources.destroy
GET    admin/{resource}/{id}/edit   arqel.resources.edit
```

Confirmações:

- **Edit, view, delete, restore (row)** mapeiam a rotas existentes — bug é só URL/method ausentes na serialização
- **Bulk delete** **não tem rota** — precisa ser adicionada
- A action sozinha **não sabe** em que resource está; o `InertiaDataBuilder` é quem tem essa info quando serializa

### Resultado de v0.9.2

A v0.9.2 corrigiu o render frontend (`actions.row → rowActions render-prop`, `actions.bulk → bulkActions ReactNode`, `selectedIds` state), mas o lado server emite payload com `url=null`. Frontend não consegue inventar a URL correcta — o ponto da `Action::toArray()` era precisamente isso.

## Objectivos

1. `Actions::edit()/delete()/view()/restore()/deleteBulk()` produzirem actions cuja serialização emite URL+method correctos quando usadas dentro de um resource
2. Adicionar rota `POST admin/{resource}/bulk/{action}` que executa bulk actions sobre records selecionados
3. Cobertura de testes Pest no PHP scope-limited (sem `pnpm test` global)
4. Re-validação mínima walkthrough no `arqel-test` (apenas BUG-VAL-002 + 003)
5. Publicar v0.9.3 em Packagist + NPM
6. Relatório curto de re-validação

## Não-objectivos

- Não mexer no frontend (`ArqelIndexPage.tsx` já está correcto — consome `action.url` quando o server passa)
- Não escrever Vitest novos (cobertura existente já lida com o caminho `if (action.url) ... else fallback`)
- Não fix BUG-VAL-005 (resource auto-discovery), BUG-VAL-006 (widget API), BUG-VAL-007 (release pipeline framework subtree), BUG-VAL-008 (doctor false positive), BUG-VAL-009 (peer deps)
- Não Playwright E2E
- Não recriar `arqel-test` — upgrade in-place
- Não introduzir novas features (apenas fixes)

## Constraint operacional: economia de recursos

Política session-level: **evitar recompilações/test runs globais**. Concretamente:

- Pest **scope-limited**: `vendor/bin/pest packages/actions/tests/Unit/StockUrlResolutionTest.php` (não `pest` sem args)
- Não `pnpm test` no monorepo todo (frontend não é tocado)

- Subagent dispatches consolidados (uma chamada por fase em vez de uma por task)
- Reusa `arqel-test` sandbox em vez de bootstrap novo
- Walkthrough manual mínimo (apenas 2 critérios, não 4)

```

  └─→ Fase 1 — Release v0.9.3 (PR + bump + tag + manual framework split)
```

```
[Actions::edit/delete/view/restore/deleteBulk] — UNCHANGED (no url/action default)
         ↓
[Action::toArray($user, $record, $resource)] ← NEW signature
    └── if url null && resource is set && action is stock convention
        → resolveStockUrl(resource, record) → [url, method]
         ↓

    └── propagates $resource to each $action->toArray(...)
         ↓
[Frontend rowActions/bulkActions render — UNCHANGED from v0.9.2]
    └── action.url is now set → router.visit(url, method) ✅

[NEW route: POST admin/{resource}/bulk/{action}]
    └── ResourceController::bulkAction
         ├── if action.hasCallback() → action.execute(Collection<Model>)
         ├── elseif action.name === 'delete' → Model::whereIn->delete (stock fallback)
         └── else → 400 with error flash
```

### Peça 1 — `Action::toArray()` aceita resource context

Modificar assinatura:

```php
public function toArray(
    ?Authenticatable $user = null,
    mixed $record = null,
    ?Resource $resource = null,
): array {
    $url = $this->resolveUrl($record);
    $method = $this->method;

    if ($url === null && $resource !== null) {
        $stock = $this->resolveStockUrl($resource, $record);
        if ($stock !== null) {
            [$url, $method] = $stock;
        }
    }

    return array_filter([
        'name' => $this->name,
        'type' => $this->type,
        'label' => $this->label,
        'icon' => $this->icon,
        'color' => $this->color,
        'variant' => $this->variant,
        'method' => $method,
        'url' => $url,
        'tooltip' => $this->tooltip,
        'disabled' => $this->isDisabledFor($record) ?: null,
        'requiresConfirmation' => $this->isRequiringConfirmation() ?: null,
        'confirmation' => $this->getConfirmationConfig(),
        'form' => $this->hasForm() ? $this->getFormSchemaArray() : null,
        'modalSize' => $this->hasForm() ? $this->getModalSize() : null,
        'successNotification' => $this->successNotification,
        'failureNotification' => $this->failureNotification,
    ], fn ($v) => $v !== null);
}

/**
 * Stock convention URLs for actions whose name matches a known
 * resource verb (view, edit, delete, restore for row; delete for
 * bulk). Returns [url, method] or null if no convention applies.
 *
 * @return array{0: string, 1: string}|null
 */
private function resolveStockUrl(Resource $resource, mixed $record): ?array
{
    $slug = $resource::$slug ?? null;
    if ($slug === null) {
        return null;
    }

    $id = is_object($record) && method_exists($record, 'getKey')
        ? $record->getKey()
        : null;

    return match ([$this->type, $this->name]) {
        ['row', 'view']    => $id !== null ? ["/admin/{$slug}/{$id}", 'GET'] : null,
        ['row', 'edit']    => $id !== null ? ["/admin/{$slug}/{$id}/edit", 'GET'] : null,
        ['row', 'delete']  => $id !== null ? ["/admin/{$slug}/{$id}", 'DELETE'] : null,
        ['row', 'restore'] => $id !== null ? ["/admin/{$slug}/{$id}/restore", 'POST'] : null,
        ['bulk', 'delete'] => ["/admin/{$slug}/bulk/delete", 'POST'],
        default            => null,
    };
}
```

Import em `Action.php`: `use Arqel\Core\Resources\Resource;`. **Nota:** `arqel-dev/actions` não pode ter hard dependency em `arqel-dev/core` por causa de ciclo. Solução: tipar como `?object $resource` no signature público + check via `method_exists` ou ducktype propriedade `$slug`. Plano detalhará.

### Peça 3 — Rota nova `POST admin/{resource}/bulk/{action}`

Localização das rotas existentes a confirmar no plano (provavelmente `packages/core/src/...routes.php` ou registadas via `ServiceProvider::boot()`).

Adicionar:

```php
Route::post('admin/{resource}/bulk/{action}', [ResourceController::class, 'bulkAction'])
    ->middleware([/* same middleware stack as siblings */])
    ->name('arqel.resources.bulk');
```

Controller method:

```php
public function bulkAction(
    Request $request,
    string $resource,
    string $action,
): RedirectResponse {
    // 1. Resolve resource (existing pattern; verify exact API in plan)
    $resourceInstance = $this->resourceRegistry->resolve($resource);

    if ($resourceInstance === null) {
        abort(404);
    }

    // 2. Find the bulk action by name on the resource's table
    $table = $resourceInstance->table();
    $bulkActions = $table?->getBulkActions() ?? [];
    $bulkAction = collect($bulkActions)
        ->first(fn ($a) => $a->getName() === $action);

    if ($bulkAction === null) {
        abort(404);
    }

    // 3. Authorize (use existing pattern from row delete)
    $this->authorize('bulk-' . $action, $resourceInstance::$model);

    // 4. Resolve records
    $recordIds = (array) $request->input('record_ids', []);
    if ($recordIds === []) {
        return back()->with('error', 'No records selected.');
    }

    $modelClass = $resourceInstance::$model;
    $records = $modelClass::whereIn('id', $recordIds)->get();

    // 5. Execute
    $hasCallback = method_exists($bulkAction, 'hasCallback')
        ? $bulkAction->hasCallback()
        : false;

    if ($hasCallback) {
        $bulkAction->execute($records, (array) $request->input('payload', []));
    } elseif ($action === 'delete') {
        // Stock fallback for deleteBulk()
        $modelClass::whereIn('id', $recordIds)->delete();
    } else {
        return back()->with('error', "Bulk action '{$action}' has no callback.");
    }

    return back()->with('success', "{$bulkAction->getLabel()} executed on " . count($records) . ' records.');
}
```

**Authorization**: a chamada `$this->authorize(...)` é placeholder. Plano detalhará a convenção real do framework (provavelmente via `Resource::canAction($user, $action, $records)` ou Laravel Policy).

## Riscos e mitigações

| Risco | Mitigação |
|-------|-----------|
| `Action.php` não pode importar `Resource` (cíclica) | Tipar como `?object $resource` + ducktype `::$slug` |
| Convenção exacta de routes/controller difere do esperado | Subagent inspeciona no início, ajusta |
| `Resource::$slug` não é `static` na classe — pode ser instance | Inspeccionar `Resource` base class; usar getter se preciso |
| Authorization na rota bulk diverge da convenção | Inspeccionar policy/gate existente para row delete; replicar |
| Test Pest no `core` requer testbench setup que pode falhar | Usar pattern de testes existentes em `packages/core/tests/Feature/*Test.php` |
| Bulk callback do user-land deve receber `Collection<Model>` mas alguns deletes fazem soft-delete vs hard | Stock fallback hardcoded a `delete()` (sem soft) — documentar; user override via `->action(Closure)` para soft |

## Convenções

- Conventional Commits + DCO sign-off
- Allowed scopes commit: `actions`, `core`, `release`, `docs`, `tickets`
- Tests scope-limited (resource economy)
- Branch protection main: PR + 3 CI checks
- Manual framework subtree split (BUG-VAL-007 pendente)
