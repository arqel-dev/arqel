## Contexto

A validação externa de v0.9.1 contra Packagist real (sandbox `~/PhpstormProjects/arqel-test/`) expôs 4 bugs reproduzíveis na UI da table que `php artisan test` não captura:

- **BUG-VAL-001** — paginação não navega após mudar perPage (e.g., 25 → 10)
- **BUG-VAL-002** — row actions (Edit, Delete) não renderizam botões na lista
- **BUG-VAL-003** — bulk action button não aparece após selecionar rows
- **BUG-VAL-004** — clear search não restaura a lista (search input vazio não invalida o filtro server-side)

Os 3 primeiros são bloqueantes da Fase 2 do roadmap; o 4º é P1. Backend, distribuição (Packagist + NPM), MCP, auth, validation, persistence, filtros, sort, theme, command palette — tudo verde.

## Diagnóstico técnico (causa raiz)

Inspecção de `packages-js/ui/src/pages/ArqelIndexPage.tsx` + `<ResourceIndex>` + `InertiaDataBuilder.php` revelou um padrão único partilhado pelos 4 bugs.

### Causa partilhada

`ArqelIndexPage` é a **ponte server-shape ↔ component-shape**. Server (via `InertiaDataBuilder::buildTableIndexData`) emite payload com:

```php
[
    'records' => [...],
    'pagination' => ['currentPage' => 1, 'lastPage' => 3, 'perPage' => 10, 'total' => 25],
    'columns' => [...],
    'filters' => [...],
    'actions' => [
        'row' => [/* ActionSchema[] */],
        'bulk' => [/* ActionSchema[] */],
        'toolbar' => [/* ActionSchema[] */],
    ],
    'search' => 'foo',
    'sort' => ['column' => ..., 'direction' => ...],
]
```

`<ResourceIndex>` (apresentacional) **não consome este shape directamente**. Espera:

- `rowActions: (record) => ReactNode` (render-prop, NOT array)
- `bulkActions: ReactNode` (rendered node, NOT array)
- `toolbarActions: ReactNode` (idem)
- handlers (`onPageChange`, `onPerPageChange`, etc.) que precisam saber **estado actual** de search/filters/perPage para emitir visits coerentes

`ArqelIndexPage` faz `<ResourceIndex {...props} ... />` (spread cego de server props) + adiciona handlers, mas **omite a tradução `actions` array → render-props** e **não tracka `perPage` em local state**, fazendo cada handler reconstruir URL com state incompleto.

### Bug-por-bug

**BUG-VAL-001 — paginação após mudar perPage:**

`buildQuery` em `ArqelIndexPage.tsx:50-62` constrói URL apenas com os params explicitamente passados em cada chamada:

```typescript
function buildQuery(params: VisitParams): Record<string, unknown> {
  // ...
  if (params.perPage) data['per_page'] = params.perPage;
  return data;
}
```

`handlePageChange` chama `buildQuery({..., page: pageNum})` sem passar `perPage`, porque `perPage` não está em local state — só ficou na URL no `handlePerPageChange` anterior. Resultado: URL após Next vai sem `?per_page=10`, backend usa default (25), retorna 25 posts em página 1, paginação parece "não navegar".

**BUG-VAL-002 e BUG-VAL-003 — row/bulk actions ausentes:**

`ArqelIndexPage.tsx` faz `<ResourceIndex {...props} filterValues={...} onFilterChange={...} ... />`. As props server contêm `actions: { row, bulk, toolbar }` arrays de `ActionSchema`. `<ResourceIndex>` recebe esses arrays mas o seu interface declara `rowActions?: (record) => ReactNode | bulkActions?: ReactNode`. React não renderiza arrays de objects — silenciosamente ignora. Result: nenhum botão aparece, sem erro de console.

**BUG-VAL-004 — clear search:**

`buildQuery` faz `if (params.search) data['search'] = params.search;`. String vazia é falsy → param omitido. Backend recebe URL sem `?search=`, mantém estado anterior porque o Inertia visit é `preserveState: true, replace: true` — backend não vê nenhum sinal para invalidar o filtro de search. Resultado: lista permanece filtrada após user apagar o input.

### Conclusão

**Os 4 bugs estão concentrados num único ficheiro** (`ArqelIndexPage.tsx`) + uso de componentes que já existem (`<ActionMenu>`, possivelmente `<BulkActionsBar>` — confirmar na implementação). Hotfix focado, baixo risco, fácil de revisar.

## Fase 0 — Hotfix UI

### Fix 1 — Track `perPage` em local state (BUG-VAL-001)

Adicionar `useState` para `perPage`, hidratar do server payload na mount, e passar em **todos** os `buildQuery({...})` calls que afectam URL:

```typescript
const [perPage, setPerPage] = useState<number | undefined>(props.pagination?.perPage);

const handlePerPageChange = (newPerPage: number): void => {
  setPerPage(newPerPage);
  visit(buildQuery({
    filters: filterValues,
    search: searchValue,
    sort: currentSort,
    perPage: newPerPage,
    // omite `page` → backend volta a 1
  }));
};

const handlePageChange = (pageNum: number): void => {
  visit(buildQuery({
    filters: filterValues,
    search: searchValue,
    sort: currentSort,
    perPage,  // <-- NEW: preserved from local state
    page: pageNum,
  }));
};

// Análogo para handleFilterChange, handleSearchChange, handleSortChange — todos passam `perPage`
```

### Fix 2 — Mapear `props.actions.row → rowActions` render-prop (BUG-VAL-002)

`ResourceIndexProps` já declara `actions: ResourceActions` (verificado em `packages-js/types/src/resources.ts:56`). Importar `<ActionMenu>` (existe em `packages-js/ui/src/action/ActionMenu.tsx`), construir render-prop:

```typescript
import { ActionMenu } from '../action/ActionMenu.js';

// dentro do componente, após hidratação de state:
const rowActions = props.actions?.row && props.actions.row.length > 0
  ? (record: TRecord) => <ActionMenu actions={props.actions.row} record={record} />
  : undefined;

// passar a <ResourceIndex>:
return <ResourceIndex<TRecord> {...props} rowActions={rowActions} ... />;
```

(Verificar API exacta de `<ActionMenu>` na implementação. Se não existir ou tiver shape diferente, fallback é renderizar `<ActionButton>` em loop directamente.)

### Fix 3 — Mapear `props.actions.bulk → bulkActions` ReactNode + selectedIds (BUG-VAL-003)

Bulk actions precisam:
1. Mapping `actions.bulk` → ReactNode rendered no toolbar
2. `selectedIds` state local (actualmente `<ResourceIndex>` recebe vazio default)

```typescript
const [selectedIds, setSelectedIds] = useState<ReadonlyArray<string | number>>([]);

const bulkActions = props.actions?.bulk && props.actions.bulk.length > 0
  ? <BulkActionsBar
      actions={props.actions.bulk}
      selectedIds={selectedIds}
      onComplete={() => setSelectedIds([])}
    />
  : undefined;

return (
  <ResourceIndex<TRecord>
    {...props}
    selectedIds={selectedIds}
    onSelectionChange={(ids) => setSelectedIds(ids)}
    bulkActions={bulkActions}
    rowActions={rowActions}
    // ... outros handlers
  />
);
```

(Confirmar se `<BulkActionsBar>` existe em `@arqel-dev/ui/action`. Se não, implementar inline com `<ActionButton>` por bulk action, dispatch via `router.post(action.url)`.)

### Fix 4 — Search vazio envia `search=` explicitamente (BUG-VAL-004)

Distinguir "search nunca foi tocado" de "search foi limpo". Solução simples: se `searchValue` vier do hook `useState` (ou seja, sempre é string definida), enviar sempre que diferente de undefined:

```typescript
function buildQuery(params: VisitParams): Record<string, unknown> {
  const data: Record<string, unknown> = {};
  const filters = pruneEmptyFilters(params.filters);
  if (Object.keys(filters).length > 0) data['filter'] = filters;
  // OLD: if (params.search) data['search'] = params.search;
  // NEW: enviar mesmo quando vazio, para limpar filtro server-side
  if (params.search !== undefined) data['search'] = params.search;
  // ... resto igual
}
```

`searchValue` no `ArqelIndexPage` já é `useState<string>(props.search ?? '')`, sempre é string definida. Mudança mínima — `params.search !== undefined` é sempre true, então `search` é sempre incluído na URL emitida (incluindo `?search=` quando vazio). Backend interpreta `?search=` (vazio) como "limpar filtro de search" — comportamento padrão do Laravel `request->input('search')` que devolve `''` em vez de manter estado anterior.

### Bonus — `selectedIds` state local

Já incluído no Fix 3. Sem este, o `enableSelection = Boolean(onSelectionChange)` em `<ResourceIndex>` é false → checkboxes nem aparecem → bulk actions não fazem sentido.

### Tests Vitest (TDD)

Criar `packages-js/ui/tests/pages/ArqelIndexPage.test.tsx`. Setup:

- Mock `@inertiajs/react`: `router.get` spy, `usePage` returning crafted props
- Render `<ArqelIndexPage>` envolvido em providers se necessários
- Usar `vitest` + `@testing-library/react` (já em uso noutros pacotes JS do monorepo)

**Casos:**

1. **Pagination preserva perPage** — render com `pagination: { currentPage: 1, perPage: 10, lastPage: 3, total: 25 }`, simular `onPerPageChange(10)` (já era 10, mas garante state) → `onPageChange(2)` → último `router.get` call deve ter `{ page: 2, per_page: 10 }`. Cobre BUG-VAL-001.

2. **Filter preserva perPage** — perPage=10 setado, simular `onFilterChange('status', 'published')` → visit deve incluir `filter[status]=published&per_page=10`. Edge case da mesma causa.

3. **Search preserva perPage** — análogo, com debounce drenado via `vi.useFakeTimers + vi.advanceTimersByTime(300)`.

4. **`actions.row` → render-prop** — render com `props.actions.row = [{ name: 'edit', label: 'Edit', ... }, { name: 'delete', label: 'Delete', ... }]`. Query DOM por `getByRole('button', { name: /edit/i })` ou similar — confirma row actions render. Cobre BUG-VAL-002.

5. **`actions.bulk` → toolbar bulk node** — render com `props.actions.bulk = [{ name: 'deleteBulk', ... }]`, simular `onSelectionChange([1, 2, 3])`, esperar bulk action button visível. Cobre BUG-VAL-003.

6. **Search clear envia `search=` vazio** — render com `search: 'foo'`, simular onChange para `''`, advance timers 300ms, assert `router.get` last call inclui `search: ''` (ou explicit empty). Cobre BUG-VAL-004.

**Edge cases extra (se houver tempo):**

- 7. `actions.row` vazio → `rowActions` é `undefined`, coluna não renderiza
- 8. `pagination` é `null` → `<TablePagination>` não renderiza
- 9. URL com filtros pré-existentes hidrata `filterValues` correctamente
