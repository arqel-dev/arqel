## Conventions

- `declare(strict_types=1)` obrigatГіrio
- Classes `final` por defeito
- **SanitizaГ§ГЈo HTML/Markdown Г© responsabilidade do consumidor** вЂ” `RichTextField` e `MarkdownField` sГі configuram o editor; o pacote NГѓO carrega `ezyang/htmlpurifier` nem qualquer parser Markdown como hard dep. Use FormRequest rules, mutators Eloquent ou o trait sanitizer planejado para FIELDS-ADV-002-followup. Para Markdown, a preview React encadeia `remark` + `rehype-sanitize`, mas qualquer renderizaГ§ГЈo server-side (blade, e-mail, RSS) deve sanitizar no boundary
- `imageUploadRoute` Г© construГ­do como string literal (`'/arqel/fields/upload?disk='.$disk`) porque a chamada `route()` exige roteamento ativo no container вЂ” manter literal preserva testabilidade unitГЎria
- Setters silenciosamente filtram entradas invГЎlidas (nГЈo-strings em `toolbar`/`customMarks`, entries sem `id`+`name` em `mentionable`) para que misconfiguraГ§Гµes no PHP nunca cheguem ao React

## Anti-patterns

- вќЊ **Persistir HTML do RichText sem purificar** вЂ” o editor produz HTML cliente-side; sempre rode `Purifier::clean($html)` (ou equivalente) num mutator/FormRequest antes de gravar
- вќЊ **`toolbar([...])` com identifiers que o `RichTextInput.tsx` nГЈo conhece** вЂ” botГµes desconhecidos sГЈo silenciosamente ignorados pelo Tiptap; consulte a lista canГіnica suportada antes de customizar
- вќЊ **Hard-dep em libs JS (Tiptap, Monaco) no `composer.json`** вЂ” dependГЄncias JS vivem em `@arqel/fields-advanced` (npm), nunca em `composer.json`
