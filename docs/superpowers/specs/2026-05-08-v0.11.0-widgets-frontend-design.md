## Contexto

A inspecção pré-spec revelou que `packages/widgets/` (PHP) já está **production-ready**: 1629 LOC distribuídas por 11 ficheiros (Widget abstract, Dashboard, DashboardRegistry, WidgetRegistry, StatWidget, ChartWidget, TableWidget, CustomWidget, controllers, routes, filters, Artisan commands, stubs). **146 Pest tests passing**. SKILL.md detalhado e canónico já escrito.

O que falta é **integração frontend**: não existe `packages-js/widgets/` workspace, não há React components a consumir o payload Inertia que `DashboardController` emite, `apps/demo` não tem um Dashboard configurado, e o metapackage `arqel-dev/framework` ainda não inclui `arqel-dev/widgets` no `require` (deixando BUG-VAL-006 ainda visível em consumidores externos).

Esta sprint **completa a integração** que falta. Não re-implementa PHP. Não cobre todos os 15 tickets do PLANNING (a maioria já estão cumpridos pelo trabalho anterior); apenas os ainda por fazer:

- **WIDGETS-010** (React components — XL no PLANNING)
- **WIDGETS-011** (DashboardGrid + layout)
- Integration: framework metapackage + demo dashboard
- Coverage: 1 spec E2E novo

WIDGETS-012 (Schedule/FullCalendar) é P2 — explicitamente não-objectivo. WIDGETS-013 (Artisan commands) já existe no PHP package.

## Objectivos

1. Criar workspace `packages-js/widgets/` com React components (Stat/Chart/Table/Custom) + DashboardGrid + DashboardFilters + WidgetWrapper + `useWidgetData` hook + `ArqelDashboardPage` Inertia entry
2. Adicionar TS types `packages-js/types/src/widgets.ts` (`WidgetSchema`, `DashboardProps`, `StatData`, `ChartData`, `TableData`, …)
3. Adicionar `arqel-dev/widgets` ao metapackage `packages/arqel/composer.json` (resolve BUG-VAL-006 a longo prazo)
4. Criar `MainDashboard` exemplo em `apps/demo/app/Arqel/Dashboards/` com 3 widgets (Stat + Chart + Table) e registá-lo no `ArqelServiceProvider`
5. Adicionar 1 spec E2E `apps/demo/tests/e2e/10-dashboard.spec.ts` ao smoke set (cobertura tornada permanente automaticamente via required CI check)
6. Publicar v0.11.0 (minor — surface pública nova: Widget API + Dashboard frontend + types)
7. Re-validar in-place no `arqel-test` sandbox

## Não-objectivos (YAGNI)

- **Não re-implementar PHP** — 146 Pest tests verdes; package está ready
- **Não Schedule/FullCalendar widget** (WIDGETS-012 P2) — backlog dedicado
- **Não cross-browser E2E** (chromium-only mantém-se)
- **Não Vitest unit tests para componentes React** — E2E cobre happy path; unit tests ficam para sprint posterior se houver dor real
- **Não Dashboard authorization scope avançado** — `canSee` no Widget já no PHP; policies complexas ficam para sprint dedicada
- **Não fix BUG-VAL-005, 008, 009, 013** — backlog regular
- **Não criar `apps/playground` separado** — `apps/demo` cumpre o papel

## Constraint operacional

Política session-level:
- Subagent dispatches consolidados onde possível (toda a Fase 1 + 2 num único subagent)
- Reusa `arqel-test` sandbox via in-place upgrade
- Pause-on-tag policy mantém-se ("C")
- Sem `pnpm test` global; scope-limited Vitest/Pest per package
- Sem rebuilds workspace-wide desnecessários (mas único pre-req do E2E run é `pnpm -r build`, esse fica)

## Arquitectura — sete fases sequenciais

```
Fase 0 — Discovery (sanity inspection of existing PHP + types + apps/demo)
  └─→ Fase 1 — packages-js/widgets workspace (config + types + React components)
       └─→ Fase 2 — Metapackage + apps/demo dashboard + E2E spec
            └─→ Fase 3 — Consolidated PR + CI watch + merge
                  └─→ Fase 4 — Release v0.11.0 (bump + tag + verify auto-publish)
                       └─→ Fase 5 — Sandbox upgrade + walkthrough
                            └─→ Fase 6 — Report + tickets advance
```

Data flow:

```
[apps/demo PHP MainDashboard]
  ├── widgets([StatWidget, ChartWidget, TableWidget])
  ├── filters([DateRangeFilter])
  └── columns(3)
        ↓ DashboardController emits Inertia payload
            { dashboard: {...}, widgets: [...] }
        ↓
[ArqelDashboardPage.tsx (packages-js/widgets)]
  ├── DashboardFilters (reusa <TableFilters> pattern)
  └── DashboardGrid
        └── per widget:
              WidgetWrapper (heading, loading state)
              └── StatWidget | ChartWidget | TableWidget | CustomWidget
                    └── useWidgetData (deferred + polling)
                          → router.reload partial OR fetch /admin/dashboards/{slug}/widgets/{id}/data
```

### Decisões técnicas

- **Recharts** como direct dep (não peer) — DX simplest
- **`DashboardGrid`** — CSS grid responsivo: `grid-template-columns: repeat(auto-fit, minmax(280px, 1fr))` + override quando widget tem `columnSpan` explícito
- **`useWidgetData`** — Inertia `router.reload({ only: [...] })` para polling/deferred fetch (server already supports both via `WidgetDataController`)
- **`<WidgetWrapper>`** — encapsula heading + description + skeleton loader quando `data === null && deferred`
- **`<ChartWidget>`** — wrapper Recharts que escolhe ResponsiveContainer + LineChart/BarChart/PieChart conforme `chartType`
- **`<DashboardFilters>`** — reusa `@arqel-dev/ui/table/TableFilters` directamente (mesma forma de filter schema)
- **`<ArqelDashboardPage>`** — espelha `<ArqelIndexPage>` pattern: `usePage`, lê `props as DashboardProps`, render `<DashboardFilters>` + `<DashboardGrid>`. Filter handlers via `router.get` com query string (mesmo padrão do v0.10.0 fix BUG-VAL-001/002 panelPath).

## Riscos e mitigações

| Risco | Mitigação |
|-------|-----------|
| Recharts SSR issues com Inertia | Inertia renderiza client-only por padrão; recharts funciona. Se SSR (futuro), `dynamic` import ou client-only branch |
| Polling sobrecarrega backend | `pollingInterval` default `null` (off); apps escolhem explicitamente |
| `DashboardGrid` layout quebra com `columnSpan` strings exóticos | Validation no PHP `Widget::columnSpan()` clamp 1..12 ou strings whitelisted (`'1/2'`, `'1/3'`, `'2/3'`, `'1/4'`, `'3/4'`, `'full'`) |
| Caret `^0.10.0` no sandbox não cobre `0.11.x` (mesmo issue v0.9 → v0.10) | Documentar bump constraint no sandbox; lesson reaproveitada |
| BUG-VAL-014 volta (code_scanning rule) | Já fixed em PR #30; PRs subsequentes mergeram clean |
| New `@arqel-dev/widgets` workspace introduz cycle import | tsconfig `composite: true` + tsup independent build evita cycles |
| `apps/demo/resources/js/app.tsx` não resolve `ArqelDashboardPage` por path mismatch | Inspeccionar `createInertiaApp` resolve callback; padrão deve incluir new path mapping |
